# OpenClaw Guard - 架构设计文档

## 概述

OpenClaw Guard 是一个生产级的 OpenClaw 防护系统，通过五层架构确保 AI 配置变更的安全性和可恢复性。

## 设计理念

**核心原则**：不要把所有安全性都压在"让 AI 遵守规则"上。AI 改崩自己的时候，恰恰是它最不可能遵守规则的时候。

**解决方案**：在 AI 之外建立独立的安全网——强制约束 + 启动守卫 + 版本控制 + 健康检查 + 远程容灾。

## 五层防护架构

```
┌─────────────────────────────────────────────────────────────┐
│  第1层：强制约束层  │  MCP 安全工具 + inotify 监控            │
├─────────────────────────────────────────────────────────────┤
│  第2层：启动守卫层  │  entrypoint-guard.sh（崩溃自动恢复）     │
├─────────────────────────────────────────────────────────────┤
│  第3层：版本控制层  │  Git 自动提交 + 增量快照                 │
├─────────────────────────────────────────────────────────────┤
│  第4层：健康检查层  │  Watchdog + 自动重启策略                 │
├─────────────────────────────────────────────────────────────┤
│  第5层：远程容灾层  │  云存储同步 + 多机热备                   │
└─────────────────────────────────────────────────────────────┘
```

### 第1层：强制约束层

**职责**：阻止非法修改，AI 无法绕过

**组件**：
- **MCP 安全修改工具** (`mcp-safe-config.py`)
  - AI 唯一允许的配置修改入口
  - 修改前强制快照
  - JSON 语法验证
  - 配置逻辑验证
  - 失败自动回滚
  - 记录修改原因

- **inotify 文件系统监控** (`inotify-watch.sh`)
  - 监控 `openclaw.json` 和 `workspace/*.md`
  - 任何修改触发即时快照
  - 系统级强制，AI 无法绕过

**恢复速度**：实时阻止（0秒）

**适用场景**：AI 试图改坏配置

### 第2层：启动守卫层

**职责**：容器启动时校验配置，损坏自动恢复

**组件**：
- **entrypoint-guard.sh**
  - JSON 语法校验
  - 配置逻辑校验（必需字段检查）
  - 健康端点预检
  - 损坏时保留崩溃现场
  - 自动恢复到 `last-known-good.json`
  - 发送告警通知

**工作流程**：
```
容器启动
  ↓
校验 openclaw.json
  ↓
[合法] → 保存为 last-known-good → 启动 OpenClaw
  ↓
[非法] → 保留崩溃现场 → 恢复 last-known-good → 发送告警 → 启动
```

**恢复速度**：10-30秒

**适用场景**：配置已损坏，容器起不来

### 第3层：版本控制层

**职责**：完整历史追踪，精确到任意时间点

**组件**：

1. **Git 自动提交**
   - 触发：每 5 分钟 / 检测到修改后
   - 命令：`git add -A && git commit -m "auto: 时间戳"`
   - 存储：完整历史 + diff
   - 回滚：`git checkout <commit>`

2. **增量快照系统** (`smart-snapshot.sh`)
   - 触发：健康检查通过后
   - 工具：`rsync --link-dest`（增量备份）
   - 频率：每小时 / 修改后
   - 保留：48小时 / 7天 / 30天
   - 清理：自动删除过期快照

**存储效率**：
- Git：50MB 目录，100 个版本可能只占 60MB
- 增量快照：只存储变化的文件，空间节省 90%+

**恢复速度**：
- Git 回滚：5秒
- 增量快照恢复：10秒

**适用场景**：
- 排查"哪次修改导致的问题"
- 回到某个时间点的完整状态

### 第4层：健康检查层

**职责**：持续监控，异常自动处理

**组件**：

1. **Docker 健康检查**
   ```yaml
   healthcheck:
     test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
     interval: 30s
     timeout: 10s
     retries: 3
   ```

2. **重启策略**
   ```yaml
   restart_policy:
     condition: on-failure
     max_attempts: 3
     delay: 10s
     window: 60s
   ```

3. **智能快照健康感知**
   - 只在 OpenClaw 健康时创建快照
   - 避免备份脏数据污染

**恢复速度**：自动重启 30-60秒

**适用场景**：运行时崩溃、进程异常退出

### 第5层：远程容灾层

**职责**：防止本地服务器彻底丢失

**组件**：

1. **实时同步（热备）**
   - 工具：`lsyncd` / `rsync`
   - 延迟：< 5秒
   - 目标：备用服务器
   - 场景：主服务器磁盘故障
   - 恢复：秒级切换

2. **定时归档（冷备）**
   - 工具：`tar.gz` + `rclone`
   - 频率：每日
   - 目标：S3/R2/OSS
   - 场景：主服务器被删/被黑
   - 恢复：5-30分钟

3. **异地多活（高级）**
   - 架构：主节点 + 多个备节点
   - 负载均衡：Nginx / Cloudflare
   - 共享存储：Redis + PostgreSQL + S3
   - 场景：区域性故障
   - 恢复：DNS 切换，用户无感知

**恢复速度**：
- 热备切换：< 10秒
- 冷备恢复：5-30分钟
- 多活切换：< 5秒（自动）

## 数据流图

### 正常修改流程

```
用户: "改配置"
  ↓
AI 解析意图
  ↓
调用 MCP: update_config()
  ↓
[第1层] 创建预修改快照 → /auto-backups/pre-change-*.json
  ↓
应用修改
  ↓
JSON 验证 → [失败] → 自动回滚 → 告警
  ↓ [成功]
[第2层] inotify 检测 → 再次快照（双保险）
  ↓
[第3层] Git 自动提交（5分钟内）
  ↓
[第4层] 健康检查通过 → 创建增量快照
  ↓
[第5层] 同步到云端/备机
  ↓
完成
```

### 崩溃恢复流程

```
配置损坏
  ↓
容器崩溃 → Docker 自动重启
  ↓
[第2层] entrypoint-guard 启动
  ↓
校验 openclaw.json → [非法]
  ↓
保留崩溃现场 → /crash-logs/openclaw.json.crash-*
  ↓
恢复 last-known-good.json
  ↓
发送告警 → 飞书/钉钉/Slack
  ↓
启动 OpenClaw
  ↓
[第4层] 健康检查通过
  ↓
恢复完成（10-30秒）
```

## 存储布局

```
/home/node/
├── .openclaw/                      # OpenClaw 主目录
│   ├── openclaw.json               # 主配置文件
│   ├── .last-known-good.json       # 上次正常配置
│   ├── workspace/                  # 工作区文件
│   ├── .git/                       # Git 版本库
│   ├── auto-backups/               # inotify 自动备份
│   │   ├── pre-modify-*.json       # 修改前快照
│   │   ├── workspace-*.md          # Workspace 快照
│   │   └── changelog.md            # 修改日志
│   └── crash-logs/                 # 崩溃现场
│       └── openclaw.json.crash-*   # 损坏的配置
│
└── openclaw-backups/               # 增量快照
    ├── 0312-1430/                  # 时间戳目录
    ├── 0312-1530/
    └── snapshot.log                # 快照日志
```

## 性能指标

| 操作 | 时间 | 空间占用 |
|------|------|---------|
| MCP 修改 + 验证 | < 1秒 | 几 KB（单个快照） |
| inotify 快照 | < 0.1秒 | 几 KB |
| Git 提交 | < 1秒 | 增量（几 KB） |
| 增量快照 | 5-10秒 | 首次全量，后续增量 |
| 启动守卫恢复 | 10-30秒 | 0（使用已有备份） |
| 一键回滚 | 1-10秒 | 0 |

## 可靠性保证

1. **多层冗余**：任何一层失效，其他层仍能工作
2. **系统级强制**：关键保护在 AI 控制之外
3. **自动化**：无需人工干预，24/7 运行
4. **可观测性**：完整日志 + 实时告警
5. **可测试性**：提供测试脚本验证各层功能

## 扩展性

- **水平扩展**：支持多机热备
- **垂直扩展**：资源限制可配置
- **插件化**：告警渠道、备份目标可扩展
- **协议兼容**：支持 MCP、REST API、CLI
